## 摘要

高吞吐量的大语言模型（LLM）服务需要同时批处理足够多的请求。然而，现有系统面临挑战，因为每个请求的键值缓存（KV cache）内存巨大且动态增长和收缩。当管理不当时，这些内存会因碎片化和冗余复制而被严重浪费，从而限制了批处理大小。为解决这一问题，我们提出了 PagedAttention，这是一种受操作系统中经典虚拟内存和分页技术启发的注意力算法。在此基础上，我们构建了 vLLM，一个 LLM 服务系统，它实现了（1）KV 缓存内存的近乎零浪费和（2）请求内部以及跨请求之间的 KV 缓存灵活共享，以进一步减少内存使用。我们的评估表明，与 FasterTransformer 和 Orca 等最先进系统相比，vLLM 在相同延迟水平下将流行 LLM 的吞吐量提高了 2-4 倍。对于更长的序列、更大的模型和更复杂的解码算法，提升效果更为显著。vLLM 的源代码公开在 https://github.com/vllm-project/vllm。

## 1 引言

像 GPT [5, 37] 和 PaLM [9] 这样的大语言模型的出现，使得编程助手 [6, 18] 和通用聊天机器人 [19, 35] 等新应用成为可能，这些应用正开始深刻影响我们的工作和日常生活。许多云公司 [34, 44] 正在竞相将这些应用作为托管服务提供。然而，运行这些应用非常昂贵，需要大量硬件加速器（如 GPU）。根据最近的估算，处理一个 LLM 请求的成本可能比传统关键字查询高出 10 倍 [43]。鉴于这些高昂的成本，提高 LLM 服务系统的吞吐量——从而降低每个请求的成本——正变得越来越重要。

LLM 的核心是自回归 Transformer 模型 [53]。该模型基于输入（提示）和迄今为止已生成的输出 token 的前序序列，一次一个地生成单词（token）。对于每个请求，这个昂贵的处理过程会重复进行，直到模型输出终止 token。这种顺序生成过程使得工作负载变为内存受限，GPU 的计算能力未被充分利用，从而限制了服务吞吐量。

通过将多个请求批处理在一起可以提高吞吐量。然而，为了批处理大量请求，每个请求的内存空间需要被高效管理。例如，图 1（左）展示了在配备 40GB RAM 的 NVIDIA A100 GPU 上，一个 13B 参数 LLM 的内存分布。大约 65% 的内存分配给模型权重，这些权重在服务过程中保持不变。近 30% 的内存用于存储请求的动态状态。对于 Transformer 来说，这些状态包括与注意力机制相关的键张量和值张量，通常称为 KV 缓存 [41]，它们表示来自早期 token 的上下文，用于按顺序生成新的输出 token。剩余的一小部分内存用于其他数据，包括激活值——评估 LLM 时创建的临时张量。由于模型权重是恒定的，且激活值只占 GPU 内存的一小部分，因此 KV 缓存的管理方式对于确定最大批处理大小至关重要。如果管理不当，KV 缓存内存在很大程度上限制批处理大小，从而限制 LLM 的吞吐量，如图 1（右）所示。

在本文中，我们观察到现有的 LLM 服务系统 [31, 60] 在高效管理 KV 缓存内存方面存在不足。这主要是因为它们将请求的 KV 缓存存储在连续的内存空间中，正如大多数深度学习框架 [33, 39] 要求张量必须存储在连续内存中一样。然而，与传统深度学习工作负载中的张量不同，KV 缓存具有独特的特性：它随着模型生成新 token 而动态增长和收缩，且其生命周期和长度事先未知。这些特性使得现有系统的方法在两方面效率显著低下：

首先，现有系统 [31, 60] 存在内部和外部内存碎片问题。为了将请求的 KV 缓存存储在连续空间中，它们根据请求的最大长度（例如 2048 个 token）预分配一块连续的内存块。这可能导致严重的内部碎片化，因为请求的实际长度可能远小于其最大长度（例如图 11）。此外，即使实际长度事先已知，预分配仍然低效：由于整个块在请求的生命周期内被预留，其他较短的请求无法利用该块中当前未使用的任何部分。除此之外，外部内存碎片也可能很严重，因为每个请求的预分配大小可能不同。事实上，我们在图 2 中的性能分析结果显示，在现有系统中，只有 20.4% - 38.2% 的 KV 缓存内存用于存储实际的 token 状态。

其次，现有系统无法利用内存共享的机会。LLM 服务经常使用高级解码算法，如并行采样和束搜索，这些算法会为每个请求生成多个输出。在这些场景中，请求由多个序列组成，这些序列可以部分共享它们的 KV 缓存。然而，在现有系统中不可能实现内存共享，因为这些序列的 KV 缓存存储在各自独立的连续空间中。

为了解决上述限制，我们提出了 PagedAttention，这是一种受操作系统解决内存碎片化和共享问题的方法——带分页的虚拟内存——启发的注意力算法。PagedAttention 将请求的 KV 缓存划分为多个块，每个块可以包含固定数量 token 的注意力键和值。在 PagedAttention 中，KV 缓存的块不必存储在连续空间中。因此，我们可以像操作系统中的虚拟内存一样以更灵活的方式管理 KV 缓存：可以将块视为页面，将 token 视为字节，将请求视为进程。这种设计通过使用相对较小的块和按需分配来减轻内部碎片化。此外，由于所有块大小相同，它消除了外部碎片化。最后，它实现了块粒度的内存共享，可以在同一请求的不同序列之间，甚至跨不同请求之间共享。

在这项工作中，我们在 PagedAttention 的基础上构建了 vLLM，一个高吞吐量的分布式 LLM 服务引擎，实现了 KV 缓存内存的近乎零浪费。vLLM 使用与 PagedAttention 协同设计的块级内存管理和抢占式请求调度。vLLM 支持流行的 LLM，如 GPT [5]、OPT [62] 和 LLaMA [52]，支持不同大小的模型，包括超过单个 GPU 内存容量的模型。我们在各种模型和工作负载上的评估表明，与最先进的系统 [31, 60] 相比，vLLM 将 LLM 服务吞吐量提高了 2-4 倍，且完全不影响模型精度。在更长的序列、更大的模型和更复杂的解码算法下，改进效果更为显著（§4.3）。

总之，我们做出了以下贡献：

- 我们识别了在服务 LLM 时内存分配面临的挑战，并量化了它们对服务性能的影响。
- 我们提出了 PagedAttention，一种注意力算法，它在非连续的分页内存上操作 KV 缓存，受操作系统中虚拟内存和分页的启发。
- 我们设计并实现了 vLLM，一个基于 PagedAttention 构建的分布式 LLM 服务引擎。
- 我们在各种场景下评估了 vLLM，并证明其显著优于之前最先进的解决方案，如 FasterTransformer [31] 和 Orca [60]。

## 2 背景

本节描述典型 LLM 的生成和服务过程，以及 LLM 服务中使用的迭代级调度。

### 2.1 基于 Transformer 的大语言模型

语言建模的任务是对 token 列表 $(x_1, ..., x_n)$ 的概率进行建模。由于语言具有自然的顺序性，通常将整个序列的联合概率分解为条件概率的乘积（即自回归分解 [3]）：

$$P(x) = P(x_1) \cdot P(x_2 | x_1) \cdots P(x_n | x_1, ..., x_{n-1}) \quad (1)$$

Transformer [53] 已经成为大规模建模上述概率的事实标准架构。基于 Transformer 的语言模型最重要的组件是其自注意力层。对于输入隐藏状态序列 $(x_1, ..., x_n) \in \mathbb{R}^{n \times d}$，自注意力层首先在每个位置 $i$ 应用线性变换以获得查询、键和值向量：

$$q_i = W_q x_i, \quad k_i = W_k x_i, \quad v_i = W_v x_i \quad (2)$$

然后，自注意力层通过将一个位置的查询向量与之前的所有键向量相乘来计算注意力得分 $a_{ij}$，并将输出 $o_i$ 计算为值向量的加权平均：

$$a_{ij} = \frac{\exp(q_i^\top k_j / \sqrt{d})}{\sum_{t=1}^i \exp(q_i^\top k_t / \sqrt{d})}, \quad o_i = \sum_{j=1}^i a_{ij} v_j \quad (3)$$

除了公式 3 中的计算外，Transformer 模型中的所有其他组件，包括嵌入层、前馈层、层归一化 [2]、残差连接 [22]、输出 logit 计算以及公式 2 中的查询、键和值变换，都以 $y_i = f(x_i)$ 的形式按位置独立应用。

### 2.2 LLM 服务和自回归生成

训练完成后，LLM 通常被部署为条件生成服务（例如补全 API [34] 或聊天机器人 [19, 35]）。对 LLM 服务的请求提供一个输入提示 token 列表 $(x_1, ..., x_n)$，LLM 服务根据公式 1 生成一个输出 token 列表 $(x_{n+1}, ..., x_{n+T})$。我们将提示和输出列表的拼接称为序列。

由于公式 1 中的分解，LLM 只能逐个采样和生成新 token，每个新 token 的生成过程依赖于该序列中所有先前的 token，具体来说是它们的键和值向量。在这个顺序生成过程中，现有 token 的键和值向量通常被缓存起来用于生成未来 token，这被称为 KV 缓存。注意，一个 token 的 KV 缓存依赖于它之前的所有 token。这意味着同一个 token 出现在序列中不同位置时，其 KV 缓存将不同。

给定一个请求提示，LLM 服务中的生成计算可分为两个阶段：

**提示阶段（Prompt Phase）**：将整个用户提示 $(x_1, ..., x_n)$ 作为输入，计算第一个新 token 的概率 $P(x_{n+1} | x_1, ..., x_n)$。在此过程中，还会生成键向量 $k_1, ..., k_n$ 和值向量 $v_1, ..., v_n$。由于提示 token $x_1, ..., x_n$ 都是已知的，提示阶段的计算可以使用矩阵-矩阵乘法操作并行化。因此，该阶段可以有效利用 GPU 固有的并行性。

**自回归生成阶段（Autoregressive Generation Phase）**：按顺序生成剩余的新 token。在迭代 $t$ 时，模型将一个 token $x_{n+t}$ 作为输入，并使用键向量 $k_1, ..., k_{n+t}$ 和值向量 $v_1, ..., v_{n+t}$ 计算概率 $P(x_{n+t+1} | x_1, ..., x_{n+t})$。注意，位置 1 到 $n+t-1$ 的键和值向量已在之前的迭代中被缓存，只有新的键和值向量 $k_{n+t}$ 和 $v_{n+t}$ 在此次迭代被计算。该阶段在两种情况下结束：序列达到最大长度（由用户指定或受 LLM 限制），或发出序列结束（\<eos\>）token。由于数据依赖性，不同迭代的计算无法并行化，且通常使用矩阵-向量乘法，效率较低。因此，该阶段严重未充分利用 GPU 计算能力，变为内存受限，占单个请求延迟的大部分。

### 2.3 LLM 的批处理技术

通过批处理多个请求可以提高 LLM 服务中的计算利用率。由于请求共享相同的模型权重，移动权重的开销在批处理中的请求之间被分摊，当批处理大小足够大时，计算开销可以覆盖该开销。然而，对 LLM 服务的请求进行批处理并非易事，原因有二。首先，请求可能在不同时间到达。一个简单的批处理策略要么让早期请求等待后期请求，要么延迟新到达的请求直到早期请求完成，导致显著的排队延迟。其次，请求可能有截然不同的输入和输出长度（图 11）。一个直接的批处理技术会对请求的输入和输出进行填充以使其长度相等，从而浪费 GPU 计算和内存。

为解决此问题，提出了细粒度批处理机制，如蜂窝批处理（cellular batching）[16] 和迭代级调度 [60]。与在请求级别运行的传统方法不同，这些技术在迭代级别运行。每次迭代后，完成的请求从批处理中移除，新的请求被加入。因此，一个新请求只需等待一次迭代即可被处理，无需等待整个批处理完成。此外，通过特殊的 GPU 内核，这些技术消除了填充输入和输出的需要。通过减少排队延迟和填充带来的低效率，细粒度批处理机制显著提高了 LLM 服务的吞吐量。

## 3 LLM 服务中的内存挑战

尽管细粒度批处理减少了计算浪费并以更灵活的方式实现了请求的批处理，但可同时批处理的请求数量仍然受到 GPU 内存容量的限制，特别是用于存储 KV 缓存的空间。换句话说，服务系统的吞吐量是内存受限的。克服这一内存限制需要解决内存管理中的以下挑战：

**大容量 KV 缓存**。KV 缓存的大小随请求数量快速增长。例如，对于 13B 参数的 OPT 模型 [62]，单个 token 的 KV 缓存需要 800KB 的空间，计算方式为 2（键和值向量）× 5120（隐藏状态大小）× 40（层数）× 2（FP16 的字节数）。由于 OPT 最多可以生成 2048 个 token 的序列，存储一个请求的 KV 缓存所需的内存量可能高达 1.6GB。当前的 GPU 内存容量为几十 GB。即使将所有可用内存分配给 KV 缓存，也只能容纳几十个请求。此外，低效的内存管理会进一步减少批处理大小，如图 2 所示。另外，根据当前趋势，GPU 的计算速度增长快于内存容量 [17]。例如，从 NVIDIA A100 到 H100，FLOPS 增加了 2 倍以上，但 GPU 内存仍停留在最大 80GB。因此，我们相信内存将成为一个越来越重要的瓶颈。

**复杂解码算法**。LLM 服务提供一系列解码算法供用户选择，每种算法对内存管理的复杂性有不同的影响。例如，当用户请求从单个输入提示生成多个随机样本时（编程建议中的典型用例 [18]），提示部分的 KV 缓存可以共享以最小化内存使用，在我们实验中该部分占总 KV 缓存内存的 12%（§6.3）。另一方面，自回归生成阶段的 KV 缓存由于不同的采样结果及其对上下文和位置的依赖性，应保持不共享。KV 缓存共享的程度取决于所使用的具体解码算法。在更复杂的算法如束搜索 [49] 中，不同的请求束可以共享更大比例的 KV 缓存（最多可节省 55% 内存，见 §6.3），且共享模式随着解码过程的推进而变化。

**未知输入和输出长度的调度**。LLM 服务的请求在其输入和输出长度上表现出变异性。这要求内存管理系统能够适应各种提示长度。此外，随着解码过程中请求的输出长度增长，其 KV 缓存所需的内存也会增加，可能耗尽可用于新请求或现有提示持续生成的内存。系统需要做出调度决策，例如从 GPU 内存中删除或交换出某些请求的 KV 缓存。

### 3.1 现有系统中的内存管理

由于当前深度学习框架 [33, 39] 中的大多数算子要求张量存储在连续内存中，之前的 LLM 服务系统 [31, 60] 也将一个请求的 KV 缓存作为连续张量存储在不同位置上。由于 LLM 的输出长度不可预测，它们基于请求的最大可能序列长度静态地为请求分配一块内存，而不考虑请求的实际输入或最终输出长度。

图 3 展示了两个请求：请求 A 的最大可能序列长度为 2048，请求 B 的最大长度为 512。现有系统中的预分配方案主要有三种内存浪费来源：
- **reserved:** 为未来 token 预留的槽位
- **internal fragmentation**: 因过度预留给可能的最大序列长度导致的内部碎片化
- **external fragmentation**: 以及来自如伙伴分配器等内存分配器的外部碎片化。
  外部碎片化永远不会用于生成的 token，这在服务请求之前是已知的。内部碎片化同样未被使用，但这只有在请求完成采样后才能意识到。它们都是纯粹的内存浪费。虽然预留的内存最终会被使用，但在整个请求持续期间保留此空间（尤其是当预留空间很大时）占据了本来可以用于处理其他请求的空间。我们在图 2 中可视化了实验中内存浪费的平均百分比，显示之前系统中实际有效内存可能低至 20.4%。

虽然已经提出了压缩 [54] 作为碎片化的潜在解决方案，但在对性能敏感的 LLM 服务系统中执行压缩是不切实际的，因为 KV 缓存数据量巨大。即使进行压缩，每个请求预分配的块空间也阻止了现有内存管理系统中特定于解码算法的内存共享。

## 4 方法

在这项工作中，我们开发了一种新的注意力算法 PagedAttention，并构建了一个 LLM 服务引擎 vLLM，以解决 §3 中概述的挑战。vLLM 的架构如图 4 所示。vLLM 采用集中式调度器来协调分布式 GPU 工作者的执行。KV 缓存管理器通过 PagedAttention 以分页方式高效管理 KV 缓存。具体来说，KV 缓存管理器通过集中式调度器发送的指令来管理 GPU 工作者上的物理 KV 缓存内存。

接下来，我们在 §4.1 中描述 PagedAttention 算法。在此基础上，我们在 §4.2 中展示 KV 缓存管理器的设计，以及在 §4.3 中它如何支持 PagedAttention。然后，我们展示这种设计如何支持各种解码方法的有效内存管理（§4.4）以及如何处理可变长度的输入和输出序列（§4.5）。最后，我们展示 vLLM 的系统设计如何在分布式设置中工作（§4.6）。

### 4.1 PagedAttention

为解决 §3 中的内存挑战，我们引入了 PagedAttention，这是一种受操作系统中经典分页思想 [25] 启发的注意力算法。与传统的注意力算法不同，PagedAttention 允许将连续的键和值存储在非连续的内存空间中。具体来说，PagedAttention 将每个序列的 KV 缓存划分为 KV 块。每个块包含固定数量 token 的键和值向量，我们将其记为 KV 块大小（$B$）。记键块 $K_j = (k_{(j-1)B+1}, ..., k_{jB})$ 和值块 $V_j = (v_{(j-1)B+1}, ..., v_{jB})$。公式 3 中的注意力计算可以转化为以下逐块计算：

$$A_{ij} = \frac{\exp(q_i^\top K_j / \sqrt{d})}{\sum_{t=1}^{\lceil i/B\rceil} \exp(q_i^\top K_t \mathbb{1} / \sqrt{d})}, \quad o_i = \sum_{j=1}^{\lceil i/B\rceil} V_j A_{ij}^\top \quad (4)$$

其中 $A_{ij} = (a_{i,(j-1)B+1}, ..., a_{i,jB})$ 是第 $j$ 个 KV 块上的注意力得分行向量。

在注意力计算过程中，PagedAttention 内核分别识别和获取不同的 KV 块。我们在图 5 中展示了一个 PagedAttention 的示例：键和值向量分布在三个块中，且这三个块在物理内存上不连续。每次，内核将查询 token（"forth"）的查询向量 $q_i$ 与一个块中的键向量 $K_j$（例如，块 0 中的键向量 "Four score and seven"）相乘，以计算注意力得分 $A_{ij}$，然后将 $A_{ij}$ 与一个块中的值向量 $V_j$ 相乘，得出最终注意力输出 $o_i$。

总之，PagedAttention 算法允许 KV 块存储在非连续的物理内存中，这为 vLLM 中更灵活的分页内存管理提供了支持。

### 4.2 KV 缓存管理器

vLLM 内存管理器的核心思想类似于操作系统中的虚拟内存 [25]。操作系统将内存划分为固定大小的页面，并将用户程序的逻辑页面映射到物理页面。连续的逻辑页面可以对应非连续的物理内存页面，从而允许用户程序像访问连续内存一样访问内存。此外，物理内存空间不需要提前完全预留，使得操作系统可以根据需要动态分配物理页面。vLLM 使用虚拟内存背后的思想来管理 LLM 服务中的 KV 缓存。在 PagedAttention 的支持下，我们将 KV 缓存组织为固定大小的 KV 块，就像虚拟内存中的页面一样。

请求的 KV 缓存表示为一系列逻辑 KV 块，从左到右填充，随着新 token 及其 KV 缓存的生成而增长。最后一个 KV 块中未填充的位置保留给未来的生成。在 GPU 工作者上，块引擎分配一块连续的 GPU DRAM 并将其划分为物理 KV 块（在 CPU RAM 上也进行类似操作用于交换；见 §4.5）。KV 块管理器还维护块表（block tables）——每个请求的逻辑和物理 KV 块之间的映射。每个块表条目记录逻辑块对应的物理块以及已填充的位置数量。分离逻辑和物理 KV 块使得 vLLM 能够动态增长 KV 缓存内存，而无需提前为所有位置预留空间，从而消除了现有系统中的大部分内存浪费，如图 2 所示。

### 4.3 使用 PagedAttention 和 vLLM 进行解码

接下来，我们通过图 6 中的示例来演示 vLLM 如何执行 PagedAttention 并在单个输入序列的解码过程中管理内存：

1. **像 OS 的虚拟内存一样**，vLLM 不需要初始时为最大可能生成的序列长度预留内存。相反，它只预留必要的 KV 块来容纳提示计算期间生成的 KV 缓存。在这种情况下，提示有 7 个 token，因此 vLLM 将前 2 个逻辑 KV 块（0 和 1）映射到 2 个物理 KV 块（分别为 7 和 1）。在预填充步骤中，vLLM 使用传统的自注意力算法（例如 [13]）生成提示的 KV 缓存和第一个输出 token。然后 vLLM 将前 4 个 token 的 KV 缓存存储在逻辑块 0 中，将接下来的 3 个 token 存储在逻辑块 1 中。剩余的一个槽位保留给后续的自回归生成阶段。

2. **在第一个自回归解码步骤中**，vLLM 使用 PagedAttention 算法在物理块 7 和 1 上生成新的 token。由于最后一个逻辑块中还有一个可用槽位，新生成的 KV 缓存存储在那里，块表的"#filled"记录被更新。

3. **在第二个解码步骤中**，由于最后一个逻辑块已满，vLLM 将新生成的 KV 缓存存储在新的逻辑块中；vLLM 为其分配一个新的物理块（物理块 3），并将此映射存储在块表中。

全局来看，对于每次解码迭代，vLLM 首先选择一组候选序列进行批处理（更多见 §4.5），并为新需要的逻辑块分配物理块。然后，vLLM 将当前迭代的所有输入 token（即提示阶段请求的所有 token 和生成阶段请求的最新 token）拼接成一个序列，输入到 LLM 中。在 LLM 计算过程中，vLLM 使用 PagedAttention 内核访问以逻辑 KV 块形式存储的先前 KV 缓存，并将新生成的 KV 缓存保存到物理 KV 块中。在一个 KV 块中存储多个 token（块大小 > 1）使得 PagedAttention 内核能够并行处理更多位置的 KV 缓存，从而提高硬件利用率并减少延迟。然而，较大的块大小也会增加内存碎片化。我们在 §7.2 中研究块大小的影响。

再次强调，vLLM 在生成更多 token 及其 KV 缓存时，动态地将新的物理块分配给逻辑块。由于所有块从左到右填充，且只有在前一个块已满时才分配新的物理块，vLLM 将一个请求的所有内存浪费限制在一个块内，因此可以有效地利用所有内存，如图 2 所示。这使得更多请求能够放入内存进行批处理——从而提高了吞吐量。当一个请求完成生成后，其 KV 块可以被释放，用于存储其他请求的 KV 缓存。在图 7 中，我们展示了 vLLM 管理两个序列内存的示例。两个序列的逻辑块映射到 GPU 工作者中块引擎预留空间内的不同物理块。两个序列的相邻逻辑块不需要在物理 GPU 内存中连续，物理块的空间可以被两个序列有效利用。

### 4.4 应用于其他解码场景

§4.3 展示了 PagedAttention 和 vLLM 如何处理基本的解码算法，如贪心解码和采样，这些算法以一个用户提示作为输入并生成单个输出序列。在许多成功的 LLM 应用 [18, 34] 中，LLM 服务必须提供更复杂的解码场景，这些场景表现出复杂的访问模式和更多的内存共享机会。本节展示 vLLM 对这些场景的通用适用性。

#### 并行采样

在基于 LLM 的程序助手 [6, 18] 中，LLM 为单个输入提示生成多个采样输出；用户可以从各种候选中选择喜欢的输出。到目前为止，我们隐含地假设一个请求生成单个序列。在本文剩余部分，我们假设更一般的情况，即一个请求生成多个序列。在并行采样中，一个请求包含多个共享相同输入提示的样本，从而允许提示的 KV 缓存也被共享。通过其 PagedAttention 和分页内存管理，vLLM 可以轻松实现这种共享并节省内存。

图 8 展示了两个输出的并行解码示例。由于两个输出共享相同的提示，我们在提示阶段只保留一份提示状态的副本；两个序列提示部分的逻辑块映射到相同的物理块：两个序列的逻辑块 0 和 1 分别映射到物理块 7 和 1。由于单个物理块可以映射到多个逻辑块，我们为每个物理块引入引用计数。在这种情况下，物理块 7 和 1 的引用计数均为 2。在生成阶段，两个输出采样不同的输出 token，需要为 KV 缓存分别存储。vLLM 在物理块粒度上实现了写时复制（copy-on-write）机制，用于需要在多个序列之间修改的物理块，类似于 OS 虚拟内存中的写时复制技术（例如，fork 进程时）。具体来说，在图 8 中，当样本 A1 需要写入其最后一个逻辑块（逻辑块 1）时，vLLM 识别到相应物理块（物理块 1）的引用计数大于 1；它分配一个新的物理块（物理块 3），指令块引擎从物理块 1 复制信息，并将引用计数减为 1。接下来，当样本 A2 写入物理块 1 时，引用计数已减少为 1；因此 A2 直接将其新生成的 KV 缓存写入物理块 1。

总之，vLLM 实现了大多数用于存储提示 KV 缓存的空间在多个输出样本之间共享，最后一个逻辑块除外，它通过写时复制机制管理。通过在多个样本间共享物理块，内存使用可以大大减少，特别是对于长输入提示。

#### 束搜索

在机器翻译 [59] 等 LLM 任务中，用户期望 LLM 输出前 k 个最合适的翻译结果。束搜索 [49] 被广泛用于从 LLM 解码最可能的输出序列，因为它减轻了完全遍历样本空间的计算复杂性。该算法依赖于束宽参数 $k$，它决定了每一步保留的 top-k 候选数量。在解码过程中，束搜索通过考虑所有可能的 token 来扩展束中的每个候选序列，使用 LLM 计算它们各自的概率，并从 $k \cdot |V|$ 个候选中保留 top-k 个最可能的序列，其中 $|V|$ 是词汇表大小。

与并行解码不同，束搜索不仅允许共享初始提示块，还允许在不同候选之间共享其他块，且共享模式随着解码过程的推进而动态变化，类似于 OS 中由复合 fork 创建的进程树。图 9 展示了 vLLM 如何管理一个 $k=4$ 的束搜索示例的 KV 块。在虚线所示的迭代之前，每个候选序列已经使用了 4 个完整的逻辑块。所有束候选共享第一个块 0（即提示）。候选 3 从第二个块开始与其他候选不同。候选 0-2 共享前 3 个块，并在第四个块处分叉。在后续迭代中，top-4 可能的候选都来自候选 1 和 2。由于原始候选 0 和 3 不再在 top 候选中，它们的逻辑块被释放，相应物理块的引用计数减少。vLLM 释放所有引用计数变为 0 的物理块（块 2、4、5、8）。然后，vLLM 分配新的物理块（块 9-12）来存储新候选的新 KV 缓存。现在，所有候选共享块 0、1、3；候选 0 和 1 共享块 6，候选 2 和 3 进一步共享块 7。

之前的 LLM 服务系统需要在束候选之间频繁地复制 KV 缓存内存。例如，在图 9 所示的情况下，虚线之后，候选 3 需要复制候选 2 的大部分 KV 缓存才能继续生成。这种频繁的内存复制开销通过 vLLM 的物理块共享大大减少。在 vLLM 中，不同束候选的大多数块可以共享。写时复制机制仅在新生 token 落在旧的共享块内时触发，类似于并行解码。这仅涉及复制一个块的数据。

#### 共享前缀

通常，LLM 用户会提供（长）任务描述，包括指令和输入输出示例，也称为系统提示 [36]。该描述与实际任务输入拼接在一起形成请求的提示。LLM 基于完整提示生成输出。图 10 展示了一个示例。此外，共享前缀可以通过提示工程进一步调整，以提高下游任务的准确率 [26, 27]。

对于这类应用，许多用户提示共享一个前缀，因此 LLM 服务提供商可以提前存储前缀的 KV 缓存，以减少花在前缀上的冗余计算。在 vLLM 中，这可以通过 LLM 服务提供商为一组预定义的共享前缀预留一组物理块来方便地实现，类似于 OS 如何在进程间处理共享库。带有共享前缀的用户输入提示可以简单地将它的逻辑块映射到缓存的物理块（最后一个块标记为写时复制）。提示阶段计算只需要在用户的任务输入上执行。

#### 混合解码方法

前面讨论的解码方法展现出多样的内存共享和访问模式。尽管如此，vLLM 支持同时处理具有不同解码偏好的请求，而现有系统无法高效做到这一点。这是因为 vLLM 通过一个通用的映射层（将逻辑块转换为物理块）隐藏了不同序列之间的复杂内存共享。LLM 及其执行内核只看到每个序列的物理块 ID 列表，不需要处理跨序列的共享模式。与现有系统相比，这种方法拓宽了对具有不同采样需求的请求进行批处理的机会，最终提高了系统的整体吞吐量。

### 4.5 调度与抢占

当请求流量超过系统容量时，vLLM 必须优先处理一部分请求。在 vLLM 中，我们对所有请求采用先来先服务（FCFS）调度策略，确保公平性并防止饥饿。当 vLLM 需要抢占请求时，它确保最早到达的请求先被服务，最晚的请求先被抢占。

LLM 服务面临一个独特的挑战：LLM 的输入提示长度变化很大，且由此产生的输出长度事先未知，这取决于输入提示和模型。随着请求数量及其输出的增长，vLLM 可能耗尽 GPU 的物理块来存储新生成的 KV 缓存。在这种情况下，vLLM 需要回答两个经典问题：（1）应该驱逐哪些块？（2）如何恢复被驱逐的块（如果需要的话）？通常，驱逐策略使用启发式方法来预测哪个块将在未来最远的时间被访问，并驱逐该块。由于在我们的情况下，我们知道一个序列的所有块是一起被访问的，我们实现了**全有或全无的驱逐策略**，即要么驱逐一个序列的所有块，要么不驱逐。此外，一个请求内的多个序列（例如，一个束搜索请求中的束候选）作为序列组一起进行组调度。一个序列组内的序列总是被一起抢占或重新调度，因为这些序列之间可能存在内存共享。

对于如何恢复被驱逐块的第二个问题，我们考虑两种技术：

**交换（Swapping）**：这是大多数虚拟内存实现使用的经典技术，它将驱逐的页面复制到磁盘上的交换空间。在我们的情况下，我们将驱逐的块复制到 CPU 内存。如图 4 所示，除了 GPU 块分配器外，vLLM 还包括一个 CPU 块分配器来管理交换到 CPU RAM 的物理块。当 vLLM 用尽空闲物理块用于新 token 时，它选择一组序列进行驱逐，并将其 KV 缓存传输到 CPU。一旦它抢占了一个序列并驱逐其块，vLLM 将停止接受新请求，直到所有被抢占的序列完成。当一个请求完成后，其块从内存中释放，被抢占序列的块被带回以继续处理该序列。注意，在此设计下，交换到 CPU RAM 的块数量永远不会超过 GPU RAM 中的总物理块数量，因此 CPU RAM 上的交换空间受限于为 KV 缓存分配的 GPU 内存量。

**重计算（Recomputation）**：在这种情况下，当被抢占的序列被重新调度时，我们简单地重新计算 KV 缓存。注意，重计算延迟可能显著低于原始延迟，因为在解码时生成的 token 可以与原始用户提示拼接成一个新提示——它们所有位置的 KV 缓存可以在一次提示阶段迭代中生成。

交换和重计算的性能取决于 CPU RAM 和 GPU 内存之间的带宽以及 GPU 的计算能力。我们在 §7.3 中检查交换和重计算的速度。

### 4.6 分布式执行

许多 LLM 的参数大小超过单个 GPU 的容量 [5, 9]。因此，有必要将它们划分到分布式 GPU 上，并以模型并行的方式执行 [28, 63]。这需要一个能够处理分布式内存的内存管理器。vLLM 通过支持 Transformer 上广泛使用的 Megatron-LM 风格张量模型并行策略 [47]，在分布式设置中非常有效。该策略遵循 SPMD（单程序多数据）执行调度，其中线性层被划分为执行逐块矩阵乘法，GPU 通过 all-reduce 操作不断同步中间结果。具体来说，注意力算子在注意力头维度上被分割，每个 SPMD 进程负责多头注意力中的一部分注意力头。

我们观察到，即使在模型并行执行下，每个模型分片仍然处理相同的输入 token 集，因此需要相同位置的 KV 缓存。因此，vLLM 在集中式调度器内设置了一个单一的 KV 缓存管理器，如图 4 所示。不同的 GPU 工作者共享该管理器，以及从逻辑块到物理块的映射。这种通用的映射允许 GPU 工作者使用调度器为每个输入请求提供的物理块来执行模型。尽管每个 GPU 工作者具有相同的物理块 ID，但工作者只为其对应的注意力头存储一部分 KV 缓存。

在每一步中，调度器首先为批处理中的每个请求准备包含输入 token ID 的消息，以及每个请求的块表。然后，调度器将这条控制消息广播给 GPU 工作者。接着，GPU 工作者开始使用输入 token ID 执行模型。在注意力层中，GPU 工作者根据控制消息中的块表读取 KV 缓存。在执行过程中，GPU 工作者使用 all-reduce 通信原语同步中间结果，无需调度器的协调，如 [47] 所述。最后，GPU 工作者将此次迭代的采样 token 发回调度器。总之，GPU 工作者不需要在内存管理上进行同步，因为它们只需要在每个解码迭代开始时随步骤输入一起接收所有内存管理信息。

## 5 实现

vLLM 是一个端到端的服务系统，具有 FastAPI [15] 前端和基于 GPU 的推理引擎。前端扩展了 OpenAI API [34] 接口，允许用户为每个请求自定义采样参数，例如最大序列长度和束宽 $k$。vLLM 引擎由 8.5K 行 Python 代码和 2K 行 C++/CUDA 代码编写。我们用 Python 开发了控制相关组件，包括调度器和块管理器，同时为关键操作（如 PagedAttention）开发了自定义 CUDA 内核。对于模型执行器，我们使用 PyTorch [39] 和 Transformers [58] 实现了流行的 LLM，如 GPT [5]、OPT [62] 和 LLaMA [52]。我们使用 NCCL [32] 进行跨分布式 GPU 工作者的张量通信。

### 5.1 内核级优化

由于 PagedAttention 引入了现有系统无法高效支持的内存访问模式，我们开发了几个用于优化它的 GPU 内核。

1. **融合重塑与块写入（Fused reshape and block write）**：在每个 Transformer 层中，新的 KV 缓存被分割成块，重塑为针对块读取优化的内存布局，然后保存在块表指定的位置。为了最小化内核启动开销，我们将它们融合到单个内核中。

2. **融合块读取与注意力计算（Fusing block read and attention）**：我们改编了 FasterTransformer [31] 中的注意力内核，使其根据块表读取 KV 缓存并即时执行注意力操作。为确保合并的内存访问，我们分配一个 GPU warp 来读取每个块。此外，我们增加了对请求批处理中可变序列长度的支持。

3. **融合块复制（Fused block copy）**：由写时复制机制触发的块复制操作可能操作在不连续的块上。如果使用 cudaMemcpyAsync API，这可能导致大量小数据移动的调用。为减轻开销，我们实现了一个内核，将不同块的复制操作批处理到单个内核启动中。

### 5.2 支持各种解码算法

vLLM 使用三种关键方法实现各种解码算法：`fork`、`append` 和 `free`。`fork` 方法从现有序列创建一个新序列。`append` 方法向序列追加一个新 token。最后，`free` 方法删除序列。例如，在并行采样中，vLLM 使用 `fork` 方法从单个输入序列创建多个输出序列。然后，它在每次迭代中使用 `append` 方法向这些序列添加新 token，并使用 `free` 方法删除满足停止条件的序列。vLLM 在束搜索和前缀共享中也采用了相同的策略。我们相信未来的解码算法也可以通过组合这些方法来支持。

## 6 评估

本节中，我们评估 vLLM 在各种工作负载下的性能。

### 6.1 实验设置

**模型和服务器配置**：我们在评估中使用了 OPT [62] 模型（13B、66B 和 175B 参数）和 LLaMA [52]（13B 参数）。13B 和 66B 是 LLM 的流行大小，如一个 LLM 排行榜 [38] 所示，而 175B 是著名的 GPT-3 [5] 模型的大小。在我们所有的实验中，我们使用 Google Cloud Platform 上配备 NVIDIA A100 GPU 的 A2 实例。详细的模型大小和服务器配置如表 1 所示。

| 模型大小 | 13B | 66B | 175B |
|---------|-----|-----|------|
| GPU | A100 | 4×A100 | 8×A100-80GB |
| 总 GPU 内存 | 40 GB | 160 GB | 640 GB |
| 参数大小 | 26 GB | 132 GB | 346 GB |
| KV 缓存内存 | 12 GB | 21 GB | 264 GB |
| 最大 KV 缓存槽位数 | 15.7K | 9.7K | 60.1K |

**工作负载**：我们基于 ShareGPT [51] 和 Alpaca [50] 数据集合成工作负载，这些数据集包含实际 LLM 服务的输入和输出文本。ShareGPT 数据集是用户共享的与 ChatGPT [35] 对话的集合。Alpaca 数据集是由 GPT-3.5 使用 self-instruct [57] 生成的指令数据集。我们对数据集进行 tokenize，并使用它们的输入和输出长度来合成客户端请求。如图 11 所示，ShareGPT 数据集的输入提示平均比 Alpaca 数据集长 8.4 倍，输出平均长 5.8 倍，且方差更大。由于这些数据集不包含时间戳，我们使用不同请求速率的泊松分布生成请求到达时间。

**基准线 1：FasterTransformer**：FasterTransformer [31] 是一个高度优化延迟的分布式推理引擎。由于 FasterTransformer 没有自己的调度器，我们实现了一个具有动态批处理机制的自定义调度器，类似于现有的服务系统如 Triton [30]。具体来说，我们根据 GPU 内存容量为每个实验设置尽可能大的最大批处理大小 $B$。调度器接收最多 $B$ 个最早到达的请求，并将批处理发送给 FasterTransformer 进行处理。

**基准线 2：Orca**：Orca [60] 是一个最先进的针对吞吐量优化的 LLM 服务系统。由于 Orca 未公开可用，我们实现了自己的 Orca 版本。我们假设 Orca 使用伙伴分配算法来确定存储 KV 缓存的内存地址。我们实现了三个版本的 Orca，基于其过度预留请求输出空间的程度：

- **Orca (Oracle)**：假设系统知道请求实际将生成的输出长度。这显示了 Orca 的上限性能，在实际中是无法实现的。
- **Orca (Pow2)**：假设系统将输出空间最多过度预留 2 倍。例如，如果实际输出长度为 25，则为输出保留 32 个位置。
- **Orca (Max)**：假设系统始终预留空间直到模型的最大序列长度，即 2048 个 token。

**关键指标**：我们关注服务吞吐量。具体来说，使用不同请求速率的工作负载，我们测量系统的**归一化延迟**，即每个请求的端到端延迟除以其输出长度的平均值，如 Orca [60]。一个高吞吐量的服务系统应该在高请求速率下保持较低的归一化延迟。对于大多数实验，我们使用 1 小时跟踪进行评估。作为例外，由于成本限制，我们为 OPT-175B 模型使用 15 分钟跟踪。

### 6.2 基本采样

我们评估 vLLM 在三个模型和两个数据集上的基本采样性能（每个请求一个样本）。图 12 的第一行展示了 ShareGPT 数据集上的结果。曲线表明，随着请求速率的增加，延迟最初以平缓的速度增加，但随后突然爆发。这可以归因于当请求速率超过服务系统的容量时，队列长度持续无限增长，请求的延迟也随之增长。

在 ShareGPT 数据集上，与 Orca (Oracle) 相比，vLLM 可以承受 1.7×–2.7× 更高的请求速率，与 Orca (Max) 相比为 2.7×–8×，同时保持相似的延迟。这是因为 vLLM 的 PagedAttention 可以高效管理内存使用，从而能够比 Orca 批处理更多的请求。例如，如图 13a 所示，对于 OPT-13B，vLLM 同时处理的请求比 Orca (Oracle) 多 2.2 倍，比 Orca (Max) 多 4.3 倍。与 FasterTransformer 相比，vLLM 可以承受高达 22× 更高的请求速率，因为 FasterTransformer 没有使用细粒度调度机制，并且像 Orca (Max) 一样低效地管理内存。

图 12 的第二行和图 13b 展示了 Alpaca 数据集上的结果，其趋势与 ShareGPT 数据集相似。一个例外是图 12 (f)，其中 vLLM 相对于 Orca (Oracle) 和 Orca (Pow2) 的优势不太明显。这是因为 OPT-175B 的模型和服务器配置（表 1）提供了大量 GPU 内存空间用于存储 KV 缓存，而 Alpaca 数据集的序列较短。在这种设置下，尽管内存管理效率低下，Orca (Oracle) 和 Orca (Pow2) 也可以批处理大量请求。结果，系统的性能变为计算受限而非内存受限。

### 6.3 并行采样和束搜索

我们评估了 PagedAttention 中内存共享在两种流行采样方法下的效果：并行采样和束搜索。在并行采样中，一个请求中的所有并行序列可以共享提示的 KV 缓存。如图 14 第一行所示，随着采样的序列数量增大，vLLM 相对于 Orca 基线带来了更大的改进。类似地，图 14 第二行展示了不同束宽下的束搜索结果。由于束搜索允许更多共享，vLLM 展示了更大的性能优势。在 OPT-13B 和 Alpaca 数据集上，vLLM 相对于 Orca (Oracle) 的改进从基本采样的 1.3× 提升到束宽为 6 时的 2.3×。

图 15 展示了内存节省量，计算方式为通过共享节省的块数除以不共享时的总块数。我们展示了并行采样 6.1% - 9.8% 的内存节省和束搜索 37.6% - 55.2% 的内存节省。在使用 ShareGPT 数据集的相同实验中，我们看到了并行采样 16.2% - 30.5% 的内存节省和束搜索 44.3% - 66.3% 的内存节省。

### 6.4 共享前缀

我们探索了 vLLM 在不同输入提示共享前缀场景下的有效性，如图 10 所示。对于模型，我们使用 LLaMA-13B [52]，它是多语言的。对于工作负载，我们使用 WMT16 [4] 英德翻译数据集，并合成了两个包含指令和几个翻译示例的前缀。第一个前缀包括单个示例（即 one-shot），而第二个前缀包括 5 个示例（即 few-shot）。如图 16 (a) 所示，当共享 one-shot 前缀时，vLLM 实现了比 Orca (Oracle) 高 1.67× 的吞吐量。此外，当共享更多示例时（图 16 (b)），vLLM 实现了比 Orca (Oracle) 高 3.58× 的吞吐量。

### 6.5 聊天机器人

聊天机器人 [8, 19, 35] 是 LLM 最重要的应用之一。为了实现聊天机器人，我们让模型通过将聊天历史记录和最后一个用户查询拼接成一个提示来生成响应。我们使用 ShareGPT 数据集合成聊天历史记录和用户查询。由于 OPT-13B 模型的上下文长度有限，我们将提示截断为最后 1024 个 token，并让模型最多生成 1024 个 token。我们不在不同的对话轮次之间存储 KV 缓存，因为这样做会占用对话轮次之间其他请求的空间。

图 17 显示，与三个 Orca 基线相比，vLLM 可以承受 2× 更高的请求速率。由于 ShareGPT 数据集包含许多长对话，大多数请求的输入提示有 1024 个 token。由于伙伴分配算法，Orca 基线为请求输出预留了 1024 个 token 的空间，无论它们如何预测输出长度。因此，三个 Orca 基线的表现相似。相比之下，vLLM 可以有效处理长提示，因为 PagedAttention 解决了内存碎片化和预留的问题。

## 7 消融研究

本节中，我们研究 vLLM 的各个方面，并通过消融实验评估我们的设计选择。

### 7.1 内核微基准测试

PagedAttention 中的动态块映射影响了涉及存储的 KV 缓存的 GPU 操作的性能，即块读/写和注意力计算。与现有系统相比，我们的 GPU 内核（§5）涉及访问块表、执行额外分支和处理可变序列长度的额外开销。如图 18a 所示，与高度优化的 FasterTransformer 实现相比，这导致注意力内核延迟增加了 20-26%。我们认为这种开销很小，因为它只影响注意力算子，而不影响模型中的其他算子，如 Linear。尽管有这种开销，PagedAttention 使 vLLM 在端到端性能上显著优于 FasterTransformer（§6）。

### 7.2 块大小的影响

块大小的选择会对 vLLM 的性能产生重大影响。如果块大小太小，vLLM 可能无法充分利用 GPU 的并行性来读取和处理 KV 缓存。如果块大小太大，内部碎片化增加且共享的概率降低。

在图 18b 中，我们使用 ShareGPT 和 Alpaca 跟踪，在固定请求速率下使用基本采样评估了 vLLM 在不同块大小下的性能。在 ShareGPT 跟踪中，块大小从 16 到 128 带来了最佳性能。在 Alpaca 跟踪中，虽然块大小 16 和 32 工作良好，但较大的块大小显著降低了性能，因为序列变得比块大小更短。在实践中，我们发现块大小 16 足够大以有效利用 GPU，又足够小以避免在大多数工作负载中出现显著的内部碎片化。因此，vLLM 将其默认块大小设置为 16。

### 7.3 比较重计算和交换

vLLM 支持重计算和交换作为其恢复机制。为了理解这两种方法之间的权衡，我们评估了它们的端到端性能并对开销进行了微基准测试，结果如图 19 所示。我们的结果显示，在较小的块大小下，交换会产生过多的开销。这是因为小块大小通常导致大量 CPU 和 GPU 之间的小型数据传输，限制了有效的 PCIe 带宽。相比之下，重计算的开销在不同块大小下保持不变，因为重计算不使用 KV 块。因此，当块大小较小时，重计算更高效；当块大小较大时，交换更高效，尽管重计算开销从未超过交换延迟的 20%。对于 16 到 64 的中等块大小，两种方法表现出可比的端到端性能。

## 8 讨论

**将虚拟内存和分页技术应用于其他 GPU 工作负载**：虚拟内存和分页的思想对于管理 LLM 服务中的 KV 缓存非常有效，因为该工作负载需要动态内存分配（因为输出长度事先未知）且其性能受 GPU 内存容量限制。然而，这并不普遍适用于每个 GPU 工作负载。例如，在 DNN 训练中，张量形状通常是静态的，因此内存分配可以提前优化。再比如，在服务非 LLM 的 DNN 时，内存效率的提高可能不会带来任何性能改善，因为性能主要是计算受限的。在这样的场景中，引入 vLLM 的技术可能会因为内存间接性和非连续块内存的额外开销而降低性能。然而，我们很高兴看到 vLLM 的技术被应用于具有类似 LLM 服务属性的其他工作负载。

**在应用虚拟内存和分页时的 LLM 特定优化**：vLLM 通过利用应用特定的语义重新解释和增强了虚拟内存和分页的思想。一个例子是 vLLM 的全有或全无交换策略，它利用处理一个请求需要其所有对应的 token 状态都存储在 GPU 内存中的事实。另一个例子是恢复被驱逐块的重计算方法，这在 OS 中是不可行的。此外，vLLM 通过将内存访问操作的 GPU 内核与其他操作（如注意力计算）的内核融合，减轻了分页中内存间接性的开销。

## 9 相关工作

**通用模型服务系统**：模型服务是近年来一个活跃的研究领域，许多系统被提出来解决深度学习模型部署的各个方面。Clipper [11]、TensorFlow Serving [33]、Nexus [45]、InferLine [10] 和 Clockwork [20] 是一些早期的通用模型服务系统。它们研究了服务单个或多个模型的批处理、缓存、放置和调度。最近，DVABatch [12] 引入了多入口多出口批处理。REEF [21] 和 Shepher [61] 提出了服务时的抢占。AlpaServe [28] 利用模型并行进行统计复用。然而，这些通用系统未能考虑 LLM 推理的自回归属性和 token 状态，从而错过了优化的机会。

**Transformer 的专用服务系统**：由于 Transformer 架构的重要性，已经开发了许多专门的 Transformer 服务系统。这些系统利用 GPU 内核优化 [1, 29, 31, 56]、高级批处理机制 [14, 60]、模型并行 [1, 41, 60] 和参数共享 [64] 来实现高效服务。其中，Orca [60] 与我们的方法最相关。

**与 Orca 的比较**：Orca [60] 中的迭代级调度和 vLLM 中的 PagedAttention 是互补的技术：虽然两个系统都旨在提高 GPU 利用率从而提高 LLM 服务的吞吐量，但 Orca 通过调度和交错请求来实现这一点，使更多请求可以并行处理，而 vLLM 通过提高内存利用率来实现，使更多请求的工作集能够放入内存。通过减少内存碎片化和启用共享，vLLM 在批处理中并行运行更多请求，与 Orca 相比实现了 2-4× 的加速。事实上，如 Orca 中那样的细粒度调度和请求交错使内存管理更具挑战性，使得 vLLM 提出的技术更加关键。

**内存优化**：加速器的计算能力和内存容量之间日益扩大的差距使得内存成为训练和推理的瓶颈。交换 [23, 42, 55]、重计算 [7, 24] 及其组合 [40] 已被用于减少训练的峰值内存。值得注意的是，FlexGen [46] 研究了在有限 GPU 内存下为 LLM 推理交换权重和 token 状态，但它不针对在线服务场景。OLLA [48] 优化张量的生命周期和位置以减少碎片化，但它不做细粒度的块级管理或在线服务。FlashAttention [13] 应用分块和内核优化来减少注意力计算的峰值内存和 I/O 成本。本文在在线服务背景下引入了块级内存管理的新思路。

---

## 10 结论

本文提出了 PagedAttention，一种新的注意力算法，允许注意力键和值存储在非连续的分页内存中，并介绍了 vLLM，一个通过 PagedAttention 实现高效内存管理的高吞吐量 LLM 服务系统。受操作系统的启发，我们展示了如何将成熟的技术（如虚拟内存和写时复制）适配用于高效管理 KV 缓存和处理 LLM 服务中的各种解码算法。我们的实验表明，与最先进的系统相比，vLLM 实现了 2-4× 的吞吐量提升。
